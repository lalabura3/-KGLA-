# PRD：AI 学习 Agent —「学知图谱」

> **文档状态**：初稿 ｜ **版本**：v0.1 ｜ **日期**：2026-04-29 ｜ **作者**：烤肠  
> **项目代号**：Knowledge Graph Learning Agent (KGLA)

---

## 1. Executive Summary

### Problem Statement
学生观看教学视频时普遍存在"看过就忘、笔记零散、知识点孤立"的问题。现有视频平台（B站、百度网盘等）虽开始提供 AI 笔记功能，但笔记仍以线性列表呈现，无法帮助学生建立知识之间的关联网络，缺乏系统性知识建构的工具。

### Proposed Solution
「学知图谱」是一个 AI 驱动的学习辅助工具，通过自动解析视频内容生成结构化图文笔记，并在知识点之间构建可交互的知识图谱，帮助学生从"被动看视频"转向"主动建构知识体系"。

### Business Impact
- 填补校园场景下"视频学习 + 知识图谱"的空白
- 可做学院教学改革亮点项目、课程思政数字化成果
- 后续可扩展到全校乃至校外，有推广价值
- 为虚拟空间设计专业提供差异化实践案例

### Timeline
| 阶段 | 周期 |
|------|------|
| 产品设计 & 审核 | 2 周 |
| MVP 开发 | 8-10 周 |
| 内部测试 & 迭代 | 2 周 |
| 试用上线 | Week 12 |

### Success Metrics
| 指标 | 目标 |
|------|------|
| 视频解析成功率 | ≥ 95% |
| AI 笔记可用率（用户不修改直接可用） | ≥ 70% |
| 知识图谱节点准确率 | ≥ 80% |
| 首批试用用户满意度 | ≥ 4/5 |

## 2. Problem Definition

### 2.1 Customer Problem

| 维度 | 说明 |
|------|------|
| **Who** | 大学生（尤其设计类/跨学科专业）、自学型学习者 |
| **What** | 看视频学习时笔记散乱，学完无法形成知识体系，复习效率低 |
| **When** | 预习阶段（不知道重点在哪）、复习阶段（找不到之前学过的内容） |
| **Where** | 宿舍、图书馆、线上课堂——任何看视频学习的环境 |
| **Why** | 视频是线性媒介，但知识是非线性的网状结构，二者存在天然矛盾 |
| **Impact** | 学习效率低下，知识留存率低，考前恶补而非日常积累 |

### 2.2 Market Opportunity

**竞争现状：**
- **百度网盘 AI 笔记**：有基础笔记功能，无知识图谱，无法跨视频关联
- **B站笔记**：功能浅，以截图+手动输入为主，AI 化程度低
- **Notion / Obsidian**：需要手动整理，无自动视频解析能力
- **ChatPPT / NotebookLM**：偏文档问答，非视频学习场景

**核心空白：**
> 市场上还没有一款 **专门为视频学习场景设计、自动构建知识图谱** 的工具。这个赛道还处于早期。

**Why Now：**
- LLM 能力成熟（语音转文字、知识提取、关系推断）
- 图数据库和可视化技术普及
- 高校对 AI+教育 的投入正在加大
- 学生群体对 AI 工具的接受度极高

### 2.3 Business Case

| 维度 | 说明 |
|------|------|
| **Strategic Value** | 学院教学数字化亮点项目，可申报教改课题 |
| **Risk Assessment** | 短期无直接竞品，但大厂（字节、百度）可能跟进，需快速 MVP 验证 |

## 3. Solution Overview

### 3.1 Proposed Solution

一款独立 Web 应用，核心链路：

```
用户导入视频 → AI 自动解析(ASR+语义分段+知识点提取) 
→ 生成图文笔记(带时间戳锚点) 
→ 构建知识图谱(自动关联+可视化) 
→ 支持探索、复习、标记掌握度
```

### 3.2 Key Differentiators

| 差异化点 | 说明 |
|----------|------|
| **知识图谱为核心** | 不是笔记的附属功能，而是知识组织的核心形态 |
| **自动关联** | AI 自动建立知识点间关系（前置/后置/包含/相似/因果） |
| **跨视频打通** | 不同视频/课程的知识点自动归并到同一张个人知识图谱 |
| **时间戳锚点** | 每条笔记/每个知识点对应视频位置，点击即跳转 |

### 3.3 In Scope (Phase 1 MVP)

| 模块 | 优先级 | 说明 |
|------|--------|------|
| 视频上传/链接导入 | P0 | 本地上传 + URL 解析 |
| 语音转文字 & 智能分段 | P0 | 基础 ASR + 语义断点 |
| AI 笔记生成（图文+时间戳） | P0 | 核心产出 |
| 单视频知识图谱 | P0 | 核心差异化 |
| 图谱节点点击查看详情 | P0 | 交互闭环 |
| 笔记手动编辑 | P1 | 提升可用性 |
| 图谱搜索/筛选 | P1 | 提升探索体验 |

### 3.4 Out of Scope (Phase 1)

- ❌ 跨视频知识图谱自动合并（Phase 2）
- ❌ 教师端 / 课程管理（Phase 2）
- ❌ 3D 知识空间（Phase 3）
- ❌ 作业/测验生成（Phase 3）
- ❌ 移动端原生 App（先做 Web 响应式）
- ❌ 多语言视频处理（先做中文）

### 3.5 MVP Definition

> **MVP 核心目标**：验证"自动视频解析 → AI笔记 → 知识图谱"这条链路对学生的实际价值。

**核心功能清单（P0）：**
1. ✅ 用户上传视频或粘贴视频链接
2. ✅ 系统自动完成 ASR + 知识点提取
3. ✅ 生成带时间戳锚点的图文笔记
4. ✅ 自动构建当前视频的知识图谱
5. ✅ 图谱节点可点击查看详情

**MVP 成功的标志：**
- 10 位学生试用，每人完成至少 3 个视频的学习
- 80% 用户表示"知识图谱有帮助理解知识点关联"
- 系统在 30 分钟内完成一段 45 分钟视频的完整处理

## 5. Design & User Experience

### 5.1 Design Principles

| 原则 | 说明 |
|------|------|
| **降低认知负荷** | 界面不堆砌信息，引导式交互，不要让用户面对空白图谱发懵 |
| **图谱优先** | 知识图谱不是辅助功能，而是核心交互界面，笔记是图谱的补充 |
| **渐进式呈现** | 不一开始就展示完整复杂图谱，从简单到复杂，用户逐步展开 |
| **学习感 > 科技感** | 不要做得像数据可视化大屏，要让用户感觉是在"学习"而非"看图表" |

### 5.2 Key Screens

**① 首页/仪表盘**
- 欢迎语 + 快速导入入口（上传/粘贴链接）
- 最近学习视频列表（卡片式）
- 个人知识图谱总览入口（缩略图模式）

**② 视频学习页（核心页面）**
布局（左右分栏，左窄右宽）：
- **左侧面板**（40%）：AI 笔记，时间戳锚点，可滚动
- **右侧面板**（60%）：视频播放器，下方图谱缩略图
- 知识点高亮联动：笔记滚动时，图谱对应节点高亮

**③ 图谱探索页**
- 全屏知识图谱
- 顶部搜索框（搜索知识点）
- 筛选面板（按掌握度/关系类型/章节）
- 点击节点 → 右侧滑出详情面板

**④ 学习记录页**
- 学习历史时间线
- 收藏夹
- 疑问/笔记汇总

### 5.3 Knowledge Graph UI Design Guidelines（给设计师的要点）

> 这是整个产品最核心的 UI/UX 难点，烤肠特别强调一下设计方向：

**要做的：**
- ✅ 图谱的交互体验参考 **Obsidian Graph View**（简洁、丝滑、无干扰）
- ✅ 节点大小和关联线粗细根据知识点重要性/关联强度变化
- ✅ 支持节点群组折叠（一个章节的知识点折叠成一个群组节点）
- ✅ 过渡动画流畅（从全景缩放到聚焦节点的过程有平滑过渡）

**不要做的：**
- ❌ 不要做成传统数据分析那种密密麻麻的带标签网络图
- ❌ 不要让节点标签在图谱上全部显示（悬停时才显示）
- ❌ 不要用闪烁/高饱和颜色
- ❌ 不要把 3D 效果强加到 MVP 版本里

**交互模式建议：**
- **默认视图**：按章节聚类，每个章节为一个群组，展开后看到该章节的知识点
- **聚焦模式**：双击某个节点，图布局重新排列，该节点居中，关联节点围绕展开
- **路径模式**：选择两个节点，图谱高亮显示最短关联路径

## 6. Technical Specifications

### 6.1 Architecture Overview

```
[Frontend - React/Next.js]
    ↓ API
[Backend - Python FastAPI / Node.js]
    ↓↓↓↓
[ASR Module] [LLM Module] [Graph Engine] [Storage]
  (Whisper)    (GPT/豆包)   (Neo4j / 内存)  (OSS + DB)
```

### 6.2 Technology Stack

| 层面 | 技术选型 | 备注 |
|------|----------|------|
| **前端框架** | React / Next.js | SSR 提升首屏性能 |
| **图谱可视化** | D3.js + vis-network（2D） / Three.js（3D 后续） | |
| **后端** | Python FastAPI | 适合 AI 任务编排 |
| **ASR** | Whisper（本地部署，利用 4×4090） | 自建算力充足，Whisper large-v3 本地跑，零 API 成本 |
| **LLM** | DeepSeek / 豆包 / Qwen（本地或 API） | 4×4090 可运行 14B-32B 开源模型本地推理；也可混合使用云端 API |
| **图数据库** | Neo4j（生产）/ JSON（MVP 先用轻量方案） | |
| **文件存储** | 服务器本地存储 / OSS | MVP 阶段先存本地，后续迁移云端 |
| **数据库** | SQLite → PostgreSQL | MVP 先用 SQLite，后续迁移 |
| **视频处理** | FFmpeg | 关键帧提取 |
| **AI 开发工具** | Cursor / Claude / Codex + GitHub Copilot | 熊哥 + AI 工具组队开发 |

### 6.3 Key Data Models

```
User
├── id, name, email
├── videos[] → Video
└── knowledge_graph → Graph (个人完整知识图谱)

Video
├── id, title, url, duration, status
├── segments[] → Segment (语义分段)
├── notes → Note (AI 笔记)
└── graph → Graph (当前视频的知识图谱)

KnowledgeNode
├── id, name, description
├── type (concept / term / formula / person)
├── timestamp (对应视频位置)
├── mastery (user's mastery level)
├── relations[] → Relation
└── source_video → Video

Relation
├── source_node → KnowledgeNode
├── target_node → KnowledgeNode
└── type (contains / prerequisite / similar / contrast / causal)
```

### 6.4 AI Pipeline

```
用户上传视频
    ↓
1. ASR 处理 → 逐字稿 + 时间轴
    ↓
2. 语义分段 → LLM 分析逐字稿，按话题划分段落
    ↓
3. 关键帧提取 → FFmpeg 按时间间隔 + 场景变化检测提取截图
    ↓
4. 知识点提取 → LLM 分析每段内容，提取知识点实体
    ↓
5. 关系推断 → LLM 分析知识点间关系，生成 Relation
    ↓
6. 笔记生成 → LLM 整理成结构化笔记（图文+时间戳）
    ↓
7. 图谱构建 → 存入图数据库，前端渲染
```

### 6.5 Security Considerations

| 问题 | 方案 |
|------|------|
| **认证** | 邮箱+验证码 / OAuth（微信/飞书登录） |
| **数据隔离** | 用户数据按 user_id 隔离存储 |
| **视频文件安全** | S3 预签名 URL，限时访问 |
| **API 防滥用** | 频率限制 + 用量配额 |
| **用户隐私** | 不上传用户学习数据用于训练 |

## 8. Risks & Mitigations

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| **ASR 准确率不足** | 中 | 高 | 预留人工修正入口；选择优质 ASR 服务商 |
| **LLM 幻觉（知识点错乱）** | 高 | 中 | 增加置信度打分，低置信度标记待人工确认 |
| **视频版权风险** | 低 | 高 | 明确用户自行上传，平台不公开分发 |
| **用户不会用图谱** | 中 | 中 | 提供默认引导流程 + 示例图谱；设计减低上手门槛 |
| **性能瓶颈（长视频）** | 中 | 中 | 异步处理队列 + 进度反馈 |
| **单人+AI开发效率瓶颈** | 中 | 中 | 善用 AI 编程工具（Cursor/Claude）极大提升效率；先做 MVP 验证再考虑扩团队 |
