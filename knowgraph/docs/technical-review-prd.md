**评审人：** Frank (技术评审员)  
**日期：** 2026-04-29  
**项目：** AI视频学习知识图谱工具

---

## 1️⃣ 技术可行性分析

### 1.1 Whisper large-v3 本地部署

**结论：30分钟内处理45分钟视频可行，但需条件约束。**

| 维度 | 分析 |
|------|------|
| GPU 需求 | Whisper large-v3 显存占用约 10GB (FP16)，4×4090 (24GB×4) 绰绰有余 |
| 吞吐评估 | 单卡 4090 上 large-v3 实时率约 20-30x（即 1分钟音频约2-3秒处理），45分钟视频音频段约 1.5-2.5分钟处理完成，完全满足30分钟窗口 |
| **前置条件** | **强烈建议先做音频分离（FFmpeg 提取音频流 → 16kHz mono WAV），避免 Whisper 直接处理视频文件** |
| 并发考量 | 若未来多用户并发上传，需实现 GPU 任务队列（Celery/Redis），单卡可同时跑2-3个推理任务 |
| 风险点 | 中文+英文混合场景下 large-v3 效果已验证，但若涉及专业术语（如医学、法律）需考虑 fine-tune 或用提示词引导 |

**推荐方案：**
- 后端：FastAPI + Celery + Redis 任务队列
- ASR 管线：FFmpeg 音轨提取 → Whisper large-v3（PyTorch + flash-attn）→ SRT/VTT 字幕输出
- 音频预处理：16kHz 重采样 + VAD（Silero VAD）切分为 30s 片段，提高并行度和准确率

### 1.2 LLM 选型

**结论：P0 阶段推荐 DeepSeek API，P1 阶段视数据量迁移至 Qwen 14B 本地。**

| 维度 | DeepSeek API | Qwen 14B 本地部署 |
|------|-------------|-------------------|
| 启动成本 | 低（开箱即用） | 高（模型部署、量化、推理优化） |
| 显存占用 | 0（云端） | ~28GB (FP16) / ~14GB (INT4)，需独占 ~1.5 张 4090 |
| 延迟 | 200-500ms/token | 50-100ms/token (本地无网络延迟) |
| 成本 | 按 token 计费 | 电费 + 算力占用 |
| 数据隐私 | 数据出站 | 完全本地 |
| MVP 适配 | ✅ 快速验证 | ⚠️ 需额外 1-2 周部署调优 |

**推荐策略：**
- **MVP 阶段：DeepSeek API** — 快速验证笔记生成、图谱构建质量，降低早期复杂度
- **P1 阶段：Qwen 14B + vLLM** — 当需离线场景、数据隐私要求提升时迁移
- **架构建议：** 后端抽象 LLM Provider 接口层，支持热切换，避免后续大规模重构

### 1.3 D3.js / vis-network 交互式知识图谱

**结论：可行，但需注意性能边界。**

| 维度 | 分析 |
|------|------|
| D3.js | 能力最强、定制度最高，但学习曲线陡峭，适合高度定制场景 |
| vis-network | 开箱即用，有物理引擎、聚类、筛选等内置功能，开发效率高 |
| **推荐组合** | **vis-network 做核心图谱渲染 + D3.js 做统计图表/时间轴**，兼顾开发速度和灵活性 |
| 性能边界 | vis-network 在 200+ 节点时可能卡顿，需实现虚拟化/分层加载 |
| Obsidian Graph View 风格 | 可通过 vis-network 的 physics solver + custom styling 实现类似效果 |

**关键性能建议：**
- 单视频知识图谱预计节点数 20-80（核心概念），无需担心性能
- 若后续扩展到跨视频全景图谱（500+ 节点），需引入：
  - 分层渲染（LOD）
  - WebGL 渲染方案（如 3d-force-graph）
  - 节点聚合/展开机制

### 1.4 技术栈整体评估

**已选技术栈（Next.js + FastAPI + Whisper + LLM + Docker）评估：**

| 评分项 | 评分 | 说明 |
|--------|------|------|
| 前后端分离 | ✅ | Next.js 前端 + FastAPI 后端，职责清晰 |
| 异步处理 | ⚠️ | FastAPI 本身支持 async，但视频处理需额外任务队列 |
| 容器化 | ✅ | Docker 部署，环境一致性有保障 |
| **缺失组件** | ❌ | 缺少数据库选型、任务队列、文件存储方案 |

**补充建议：**
- **数据库：** PostgreSQL（元数据/用户/笔记） + Neo4j/RedisGraph（知识图谱存储与查询）
- **任务队列：** Celery + Redis（视频处理、ASR 异步任务）
- **文件存储：** 本地文件系统（MVP）+ MinIO（可扩展至对象存储）
- **WebSocket：** 用于实时推送 ASR/笔记生成进度

## 2️⃣ MVP 范围确认

### P0 评估

| 功能 | 技术可行性 | 风险等级 | 建议 |
|------|-----------|---------|------|
| 视频上传/链接导入 | 低风险 | 🟢 | 需加文件大小/格式校验 |
| ASR + 分段 | 低风险 | 🟢 | VAD 分段策略需验证 |
| AI笔记(图文+时间戳) | 中风险 | 🟡 | Prompt Engineering 是关键 |
| 单视频知识图谱 | 中风险 | 🟡 | 图谱构建准确率决定体验天花板 |
| 节点点击详情 | 低风险 | 🟢 | 属于前端交互范畴 |

### P1 评估

| 功能 | 技术可行性 | 备注 |
|------|-----------|------|
| 笔记编辑 | 低风险 | 需考虑多人协作冲突？ |
| 图谱搜索 | 低风险 | 全文搜索 + 图遍历 |
| AI问答 | 中风险 | RAG 架构 → 依赖向量数据库 |
| 掌握度标记 | 低风险 | 用户元数据，前端状态管理 |
| 学习仪表盘 | 低风险 | 数据聚合 + 图表 |

### 建议补充/调整

1. **建议新增 P0：VAD 分段预览与手动调整** — 用户上传视频后，ASR 分段结果若偏差会连锁影响笔记和图谱质量，应允许快速纠偏
2. **建议新增 P0：导出功能（Markdown/PDF）** — 这是学习场景的核心闭环，笔记和图谱不能只看不能导出
3. **建议降级 P1：视频链接导入** — 如涉及 YouTube/Bilibili 等外部平台，可能涉及 API 变更、版权、反爬等不可控因素。建议 P0 只支持本地视频上传，链接导入移至 P1
4. **P1 AI问答 建议明确定义范围** — "基于本视频内容的问答" vs "跨视频知识库问答"，后者需要向量数据库 + 跨视频索引，差异巨大

## 3️⃣ UI/UX 方向 — 技术评估

| 需求 | 技术可行性 | 建议 |
|------|-----------|------|
| Obsidian Graph View 风格 | ✅ | vis-network physics + dark theme 可实现 |
| 章节聚类 | ✅ | 通过时间戳分组 + force-directed clustering |
| 聚焦模式 | ✅ | 双击节点 → camera focus + 子图展开 |
| 路径模式 | ✅ | Dijkstra/BFS 最短路径高亮，简单图算法 |
| 响应式设计 (PC+平板) | ✅ | Tailwind CSS + 响应式断点，vis-network 支持 touch events |
| **不足** | ⚠️ | 未提及无障碍 (a11y)、国际化、暗色/亮色主题切换 |

**技术建议：**
- 图谱渲染用 Canvas 模式（vis-network 默认），移动端性能优于 SVG
- 节点颜色编码需设计清晰的语义映射（概念类型 → 颜色），并支持色盲友好模式

## 4️⃣ 风险与缓解 — 技术视角

| 风险 | 严重程度 | 技术缓解方案 |
|------|---------|-------------|
| **ASR 准确率** | 🟡 中 | 1) VAD 精细化分段提升上下文连贯性 2) 提供手动修正界面 3) 专业术语词典注入 Whisper prompt 4) 中英混合场景用 language detection 预处理 |
| **LLM 幻觉** | 🔴 高 | 1) 笔记生成强制要求引用时间戳/原文 2) 图谱关系必须来自原文证据 3) 置信度评分+低置信度项标记 4) 输出结构化 JSON 约束格式（减少自由文本） 5) 用户反馈机制（标记错误 → 持续改进） |
| **视频版权** | 🟡 中 | 纯技术工具，用户自行负责内容版权合规 → 在用户协议和上传界面做免责声明 |
| **用户上手门槛** | 🟡 中 | 1) 首次使用引导教程（onboarding flow） 2) 示例视频+图谱 Demo 3) 关键操作 tooltip |
| **GPU 资源争抢** | 🟡 中 | 任务优先级队列，ASR > LLM > 图谱构建，避免多任务同时抢占 GPU |
| **大文件上传** | 🟢 低 | 分片上传 + 断点续传，45分钟 1080p 视频约 2-4GB，需前端进度条 |

## 5️⃣ 开发时间线评估

**MVP 8-10 周评估：时间紧凑但在合理范围。**

建议的阶段划分：

| 周次 | 阶段 | 关键交付 |
|------|------|---------|
| W1-2 | 基础设施搭建 | Docker 环境、FastAPI 骨架、Next.js 骨架、DB schema、任务队列 |
| W3-4 | ASR 管线 | FFmpeg 集成、Whisper 部署、VAD 分段、WebSocket 进度推送 |
| W5-6 | AI 笔记生成 | LLM 接入、Prompt Engineering、笔记展示界面 |
| W7-8 | 知识图谱 | 图谱构建算法、vis-network 集成、交互功能（聚焦/路径/详情） |
| W9-10 | 联调打磨 | 全链路测试、性能优化、错误处理、部署文档 |

**关键里程碑风险点：**
- W3-4：Whisper 部署调优可能超时（CUDA/cuDNN 版本兼容），需预留缓冲
- W5-6：Prompt 迭代不可预测，建议提前准备 10+ 测试视频建立评估基准
- W7-8：图谱交互复杂度可能超预期（拖拽、缩放、聚类、动画），建议尽早出低保真原型验证

## 📋 总结与建议

### 立即行动项（本周）

1. **确认数据库选型** — PostgreSQL + 图谱存储方案（Neo4j vs RedisGraph vs pgRouting）
2. **搭建 Docker 开发环境** — 统一 Node.js/Python/CUDA 版本
3. **准备测试视频集** — 5-10 个不同学科/时长/语速的视频，用于 ASR + LLM 效果评估
4. **GPU 环境验证** — 确认 4090 服务器 CUDA 版本、驱动版本、Whisper 兼容性

### 架构红线（不可妥协）

1. **LLM Provider 必须抽象接口化** — 保证 DeepSeek → Qwen 切换成本最小
2. **ASR 输出必须保存原始结果** — 后续修正/重跑都依赖原始数据
3. **所有异步任务必须有状态追踪** — Pending → Processing → Done/Failed，前端可轮询
4. **API 必须版本化** (`/api/v1/...`) — 为后续迭代留余地
